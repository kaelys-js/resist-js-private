## `.claude/lib/baseline-compare.mjs`

128-line Node script (`#!/usr/bin/env node`) called by `.claude/hooks/post-edit-format-lint.sh`.

**Purpose**: compares post-edit lint findings against a per-file baseline (`.claude/lint-baseline.json`) so existing diagnostics don't block new edits — only NEW findings introduced by the edit are flagged.

**Interface**:
- `process.argv[2]` — JSON string with `resist-lint --json` output.
- Env vars (required):
  - `BASELINE_PATH` — path to `lint-baseline.json` (count-map format, key = `${file}|${ruleId}|${message}`).
  - `EDITED_FILE` — absolute path of the file that was just edited.

**Output on stdout**:
- `BLOCK\n<findings...>` — new findings detected (script exits 0; caller decides).
- `SHRUNK <N>` — baseline auto-shrunk by N (count went down on this edited file).
- `NOOP` — no findings, nothing to shrink.

**Algorithm**:
1. Parses current findings from stdin JSON.
2. Loads `lint-baseline.json` (rejects legacy array format → forces regen).
3. Builds `currentCounts: Record<key, number>` and `samples: Record<key, message>` scoped to `EDITED_FILE`.
4. Walks baseline keys owned by `EDITED_FILE`:
   - Current count > baseline → `BLOCK` (new findings introduced).
   - Current count < baseline → `SHRUNK` (auto-update baseline DOWN; never UP).
   - Equal → no change.
5. New keys not in baseline → `BLOCK`.
6. On `SHRUNK`, overwrites `lint-baseline.json` with reduced counts. **Only touches keys owned by EDITED_FILE.** Never increases counts.

**Why this design**: Lets the linter ratchet downward — every time you fix a diagnostic in a file, the baseline tightens. Existing diagnostics in OTHER files stay tracked but don't block your current edit. New diagnostics in the edited file always block.

**Caller**: `post-edit-format-lint.sh` runs the linter on `EDITED_FILE`, pipes the JSON into `node .claude/lib/baseline-compare.mjs`, reads first line, decides whether to emit a system-reminder back to Claude.
